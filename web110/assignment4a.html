The first activity I had chosen to do was to complete three tutorials on internet reference models, internet protocols, and internet Domain Name System. I had learned quite a bit about the back-end of the internet and how information is passed to and from your computer. This is done using the TCP/IP protocol, the computer sends and recieves packets. I had learned that the protocol isn't guaranteed to send or recieve packets. Instead, the packets can get lost which, in turn makes your connection lag or not work at all. I also learned how the UDP protocol works. Before hand, I had messed with it due to tutorials to help with my own connection, but had never really looked into how it works. For example, UDP doesn't require a connection with the host to exchange data. However, this is also an unreliable protocol so the data has no guarantee that it will be recieved. The last thing I had learned about the UDP protocol is that it is used to send small amounts of data at a time instead of large amounts of data or packets. I can use this in my own life whenever I need to explain to someone how the internet works while trying to diagnose their connection issues.
The second activity I had chosen to do was verifying your internet connectivity. I had some experience with this already from troubleshooting my own internet issues over the years. However, I had never used the website traceroute-online.com to visually trace the route to a website. I had learned that you can visually trace the route your computer takes using a website. I had also learned that when using the nslookup command inthe command prompt to display the DNS server IPs of a website, you can essentially filter which IP protocol version you are looking for by typing ipv4. or ipv6. before the website URL. I can use this when diagnosing my own internet issues as well as friends, family, and clients' internet issues as well. It gives a deeper look into what is going on and can give a vague idea of the issue the internet is having when connecting to the internet.
